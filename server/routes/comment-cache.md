# Next.js 缓存机制 Q&A

## 1. `default` 字符串是什么意思？

在 Next.js 的较新版本（尤其是你使用的 16.1.6 版本，它引入了更细粒度的缓存控制）中，`revalidateTag` 的第二个参数 `profile` 是用来指定**缓存策略（Cache Profile）**的。

简单来说，`"default"` 字符串的意思如下：

1.  **指定策略名称**：Next.js 提供了一系列预定义的缓存策略（如 `"default"`, `"seconds"`, `"minutes"`, `"hours"` 等）。这些策略定义了数据在缓存中可以存在多久、什么时候变旧（stale）以及什么时候失效。
2.  **默认行为**：`"default"` 是最常用的标准策略。它告诉 Next.js 使用全局默认的缓存生命周期设置来处理这个标签的失效。
3.  **约束关系**：在你的代码中，`getCommentsCached` 使用了 `unstable_cache`。当你调用 `revalidateTag(tag, "default")` 时，它是为了确保匹配该标签的所有缓存项按照 `"default"` 策略的要求进行重新验证。

### 为什么必须传它？

在旧版本中，`revalidateTag` 只需要一个标签名。但在新版本中，为了支持更复杂的缓存场景（比如有些数据缓存几秒，有些缓存几天），Next.js 要求明确指出你要触发哪种策略下的失效。

如果你想查看具体的策略定义，可以在 `node_modules/next/cache.d.ts` 中看到类似这样的预设：

- **`default`**: 适用于大多数场景。
- **`seconds`**: 较短的缓存时间（约 1 秒后开始后台重新验证）。
- **`minutes`**: 缓存分钟级别。

总之，在这里使用 `"default"` 是最标准、最稳妥的做法，意为“按照默认的缓存规则使该标签失效”。

## 2. `unstable_cache` 封装是什么意思，然后再用 cache 封装。 为什么这么做？

这是一个非常经典且重要的 Next.js 缓存性能优化模式。简单来说，这是**“跨请求缓存”**与**“同一次请求内去重”**的组合拳。

我们可以把它们的作用拆解开来看：

### 1. `unstable_cache` (来自 `next/cache`)：跨请求的“硬盘”级缓存

- **作用范围**：全局。不管是谁访问，也不管访问多少次，只要缓存没过期（或者没被手动失效），大家拿到的都是同一份数据。
- **存储位置**：它可以被认为是持久化的（存储在服务器内存、文件系统或 Redis 中）。
- **核心功能**：避免频繁查询数据库。它支持 `tags`，所以你可以通过 `revalidateTag` 精确地告诉它：“评论更新了，把这个帖子的缓存删掉”。
- **比喻**：就像书店里的某本书，打印出来后就放在架子上，谁来买都是这一版，不用每次有人买都去重新印刷。

### 2. `cache` (来自 `react`)：单次请求的“内存”级记忆

- **作用范围**：仅限**当前这一次**服务器请求。当这封回给浏览器的信件处理完，这个缓存就销毁了。
- **存储位置**：当前请求的执行上下文中（内存）。
- **核心功能**：**请求去重（Memoization）**。在渲染同一个页面时，如果你在 `Page.tsx` 里调了一次 `getComments`，在底下的 `CommentList.tsx` 组件里又调了一次，`cache` 会确保真正的逻辑只执行**第一次**，第二次直接从这次请求的内存里拿结果。
- **比喻**：就像你今天去书店买书，当你还没离开书店时，如果你发现需要两本，第二次买的时候你脑子里记得刚才已经买过一本了。

---

### 为什么要“套娃”封装？（即：为什么要两者结合）

如果你仔细看代码：

```typescript
const getCommentsCached = (postId, parentId) => unstable_cache(...)();
const getCommentsMemoized = cache(getCommentsCached);
```

**原因如下：**

1.  **性能最优化**：虽然 `unstable_cache` 很快，但它依然涉及到生成缓存 Key、检查缓存过期、甚至从文件/Redis 读取数据的过程，这依然有微小的开销。
2.  **避免重复逻辑开销**：在同一次页面加载中，如果你的组件树里有 3 个地方都调用了获取评论的方法，如果没有外面那层 `cache`，Next.js 会去 `unstable_cache` 的存储库里查询 3 次（虽然 3 次都中缓存，但还是查了 3 次）。
3.  **有了 `cache` 之后**：
    - **第一次调用**：触发 `cache` -> 进入 `unstable_cache` -> 读存储/数据库 -> 返回结果 -> **存入请求内存**。
    - **第二次调用**（同一个页面渲染中）：触发 `cache` -> **直接从内存返回结果**。完全跳过了 `unstable_cache` 的内部逻辑。

### 总结

- **`unstable_cache`** 是为了保护**数据库**（别老去查表）。
- **`cache`** 是为了保护 **`unstable_cache`**（别在同一个页面里重复运行缓存检查逻辑）。

这种“两层缓存”配合 `revalidateTag` 的机制，是目前 Next.js 性能表现最顶级的写法。

## 3. 缓存key, 缓存标签，分别有什么作用？

在 `unstable_cache` 中，**缓存 Key (Cache Key)** 和 **缓存标签 (Cache Tags)** 扮演着完全不同的角色：一个是用来“存取”的唯一标识，另一个是用来“清理”的分组标签。

我们可以用书店存书的例子来类比：

---

### 1. 缓存 Key (Cache Key) —— **“书的唯一索引号”**

- **作用**：决定缓存的**唯一性**和**存取路径**。
- **代码体现**：`[`comments-${postId}-${parentId ?? "all"}`]`
- **核心逻辑**：
  - Next.js 会根据这个 Key 去缓存库里找数据。
  - 如果参数（如 `postId`）变了，生成的 Key 就不同，缓存也就互不干扰。
  - **1 对 1 关系**：一个独立的 Key 对应一份独立的数据内存。
- **比喻**：就像图书馆里每本书都有一个唯一的“索书号”。你告诉管理员“我要 A 柜子第 3 排第 5 本”，这个地址就是 Key，它是用来**定位**那一本具体的书。

### 2. 缓存标签 (Cache Tags) —— **“书的分类标签”**

- **作用**：用来**标记分组**，方便后续**批量清理**。
- **代码体现**：`tags: ["comments", `comments-post-${postId}`]`
- **核心逻辑**：
  - 标签不参与查找过程，你不能通过标签直接读取数据。
  - 它是给缓存打上的“备注”。
  - **多对多关系**：同一份缓存可以打多个标签（比如既属于“所有评论”，也属于“帖子 A 的评论”）；而同一个标签也可以标记多份缓存。
- **比喻**：就像你在书上贴了标签——“科幻”、“新书”。当你发现“科幻”类别的书信息有误，你大喊一声“清理掉所有‘科幻’标签的书！”，管理员就会把所有带这个标签的书全扔掉。

---

### 为什么在你的代码里两个都要用？

看这段逻辑：

```typescript
unstable_cache(
  async () => { ... },
  [`comments-${postId}-${parentId}`], // <--- Cache Key
  {
    tags: ["comments", `comments-post-${postId}`], // <--- Cache Tags
  }
)
```

1.  **查询时 (Query)**：当你传入 `postId="123"` 和 `parentId="abc"` 时，由于 **Key** 是唯一的，它能精准定位到“帖子123下的某个回复分支”的缓存。
2.  **失效时 (Invalidation)**：当你发表了一个新评论。你不需要知道所有分页或所有分支的 **Key** 是什么。你只需要执行：
    ```typescript
    revalidateTag(`comments-post-${postId}`, "default"); // <--- 使用 Tag 批量清理
    ```
    这一句指令，就能把**所有**打上了这个“帖子ID”标签的缓存（无论它是哪个父节点的回复、第几页）全部一次性清除。

### 总结对照表

| 特性         | 缓存 Key (Key)               | 缓存标签 (Tag)             |
| :----------- | :--------------------------- | :------------------------- |
| **主要目的** | **找**数据（确定唯一路径）   | **删**数据（确定管理分组） |
| **唯一性**   | 必须由所有参数组成，保证唯一 | 可以重复，支持分组         |
| **调用频率** | 每次读取缓存时都会用到       | 仅在需要使缓存过期时用到   |
| **对应关系** | Key -> 缓存内容 (1:1)        | Tag -> 缓存内容 (1:N)      |

通过这种设计，你既能保证全国各地的用户都能快速读取到自己看的那一部分数据，又能在一篇帖子更新时，一键刷新该帖子下的所有相关缓存。

## 4. `unstable_cache` 是不是存到服务器的内存了？

这是一个非常好的观察。答案是：**它不只是存入内存，它的目标是“持久化”存储。**

我们可以把 Next.js 的缓存存储位置分为三层来理解：

### 1. 默认情况（本地开发/简单部署）

当你运行 `npm run dev` 或在普通服务器上运行 `next start` 时：

- **存储位置**：服务器的**文件系统**（硬盘）。
- **具体路径**：通常在项目目录下的 `.next/cache/fetch-cache`（或者类似的子目录）中。
- **特点**：即使你重启了服务器进程，这些缓存依然可能存在（除非你删除了 `.next` 文件夹）。这和普通的内存变量（重启就丢）是有本质区别特。

### 2. 生产环境（如部署在 Vercel 或容器中）

- **存储位置**：**分布式缓存存储**。
- **原因**：在生产环境下，通常不只有一个服务器（会有多个副本/实例）。如果存到某个服务器的内存或本地硬盘，其他服务器就拿不到。因此，Next.js 会把 `unstable_cache` 的数据同步到一个外部存储（类似于 Redis 或高效的对象存储）。

### 3. 与 React `cache` 的对比（非常重要）

这是最容易混淆的地方：

| 缓存类型                       | 存储位置               | 生命周期                                                                      |
| :----------------------------- | :--------------------- | :---------------------------------------------------------------------------- |
| **`cache` (React)**            | **服务器内存**         | **极短**。仅存在于当前那点“发信”的过程中，信发完了，内存就放掉了。            |
| **`unstable_cache` (Next.js)** | **文件系统或外部存储** | **长**。跨越多次请求，甚至跨越服务器重启，直到通过 `revalidateTag` 主动删除。 |

---

### 总结：

`unstable_cache` **不是**简单的内存存储。

- 如果它只存内存，那么当你开启 10 个服务器实例时，数据就乱套了。
- 如果它只存内存，那么由于 Node.js 的内存限制，缓存多了会导致服务器崩溃（OOM）。

所以，`unstable_cache` 的底层是通过**序列化（JSON）**把你的数据存到了更稳妥的地方（硬盘或缓存系统），而 React 的 `cache` 则是真正存在 Node.js 的**运行内存**里。
